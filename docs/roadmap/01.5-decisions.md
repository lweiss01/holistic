## Task 1: Implicit Resume on MCP Connect

### Q1.1: Notification Strategy
**Decision:** Both visible message AND tool availability

**Implementation:**
- Send notification on connect with resume context
- Keep `holistic_resume` tool available for explicit refresh
- First connect → Full visible message
- Reconnect → Shorter notification with diff
- Tool always available for agent to call

### Q1.2: Multiple Clients
**Decision:** Each client gets resume context independently

### Q1.3: Reconnect Behavior
**Decision:** Send diff of what changed since last connect

**Example:**
```
📋 Resumed session (3 new checkpoints since last connect)
  • Checkpoint: "Added auto-tuning for file watch"
  • Checkpoint: "Implemented worktree sync"
  • Checkpoint: "Fixed merge conflict handling"
```

## Task 2: Auto-Session Inference

### Q2.1: Inference Sources
**Decision:** Check multiple signals in priority order:
1. Pending work items
2. Last handoff
3. **Recently modified files** (NEW)
4. **Open issues/PRs from repo** (NEW)
5. Git commits
6. Generic fallback

### Q2.2: Git Commit Analysis
**Decision:** Parse last N commits (not just HEAD) to find better goal

**Example:**
```typescript
// Look at last 5 commits, find common theme:
// "feat: add auto-sync" 
// "fix: sync conflict handling"
// "test: sync integration tests"
// → Inferred goal: "Working on auto-sync feature"
```

### Q2.3: Disable Auto-Inference
**Decision:** Support both one-time override AND permanent preference

**One-time:**
```bash
holistic start-new --goal "New feature"  # Explicit goal = skip inference
holistic resume --fresh                   # Prompt for goal
```

**Permanent:**
```json
// .holistic/config.json
{ "autoInferSessions": false }
```

## Task 3: Smarter Passive Checkpoints

### Q3.1: Configurable Thresholds with Auto-Tuning
**Decision:** Smart defaults with auto-tuning, optional manual override

**Default behavior:**
```json
{
  "checkpoints": {
    "fileWatch": {
      "minFiles": 3,          // Start here
      "debounceSeconds": 10
    },
    "idleMinutes": 5
  }
}
```

**Auto-tuning (learns from project):**
```typescript
// After collecting checkpoint history:
// "This repo averages 2 files per commit → adjust minFiles to 2"
// "Big refactors common → increase to 5"
```

**Manual override:**
```bash
holistic config set checkpoints.fileWatch.minFiles 5
```

### Q3.2: Checkpoint Summaries
**Decision:** Yes, capture summary of what changed

**Example:**
```json
{
  "reason": "auto-save-cluster",
  "summary": "3 files changed: src/core/state.ts, src/cli.ts, README.md",
  "changedFiles": ["src/core/state.ts", "src/cli.ts", "README.md"]
}
```

### Q3.3: Idle Status Updates
**Decision:** Yes, update `latestStatus` with idle detection

**Example:**
```
Status: "Idle - last active 5m ago (was: Completed Task 5 sync design)"
```

### Q3.4: Git Hooks and Session State
**Decision:** Git hooks have direct access to state.json (no daemon dependency)

**Architecture:**
```bash

# CLI loads state.json directly
# ✓ Works on mobile, servers, anywhere git works
# ✓ Daemon is optional enhancement, not required
```

## Task 5: Automatic State Sync

### Q5.1: Non-Disruptive Background Sync
**Decision:** Use git worktree to avoid branch switching

**Implementation:**
```bash
# One-time setup during `holistic init`:
git worktree add .holistic/.state-sync holistic/state

# During auto-sync (background worker):
# 1. Copy state to worktree
cp .holistic/state.json .holistic/.state-sync/
cp -r .holistic/context .holistic/.state-sync/

# 2. Commit and push from worktree (independent git process)
cd .holistic/.state-sync
git add .
git commit -m "chore: auto-sync holistic state"
git push origin holistic/state
```

**Benefits:**
- ✓ Zero disruption (no branch switching in main repo)
- ✓ Works with dirty working tree
- ✓ Background process doesn't block user
- ✓ Completely independent git operations

### Q5.2: Smart Merge Strategy
**Decision:** Timestamp-first with checkpoint count tiebreaker

**Algorithm:**
```typescript
function mergeStates(local: HolisticState, remote: HolisticState): HolisticState {
  const localTime = new Date(local.updatedAt).getTime();
  const remoteTime = new Date(remote.updatedAt).getTime();
  
  // 1. If one is clearly newer (>5min gap), use that one
  if (localTime - remoteTime > 300000) return local;
  if (remoteTime - localTime > 300000) return remote;
  
  // 2. If close in time, use checkpoint count as tiebreaker
  if (local.activeSession && remote.activeSession) {
    if (local.activeSession.checkpointCount > remote.activeSession.checkpointCount) {
      return local;  // More work done locally
    }
  }
  
  // 3. Default: keep remote (it synced first)
  return remote;
}
```

**Safety net:** Save backup before overwriting:
```bash
.holistic/state.json.backup-2026-03-20T19-30-00
```

### Q5.3: Branch Cleanup
**Decision:** Yes, squash/rebase to keep holistic/state branch clean

**Strategy:**
- Squash commits periodically (e.g., daily or weekly)
- Keep clean audit trail without 1000s of sync commits
- Force-push is safe (not a code branch)

### Q5.4: Sync Failure Handling
**Decision:** Yes, create pending work item on sync failures

**Example:**
```
⚠️  Pending Work: Sync failing - check network connection
    Last successful sync: 2 hours ago
    Error: Network timeout on push to origin/holistic/state
```

## Task 7: General Architecture

### Q7.1: Process Architecture
**Decision:** Separate processes with clear roles

**MCP Server (client-controlled):**
```bash
holistic serve
  - Started by Claude Desktop/Cursor/etc.
  - Handles tool calls from agents
  - CAN do file watching while connected (optional)
  - Stops when client disconnects
```

**Daemon (user-controlled, optional):**
```bash
holistic daemon
  - Background file watcher (always running)
  - Independent of MCP connections
  - Fills gaps when no MCP client active
  - Optional (git hooks work without it)
```

**Coordination:** Both use state.json with file locking. If MCP server is running, daemon defers to it.

### Q7.2: State Locking
**Decision:** Phase 1 concurrency protection is sufficient, but keep race conditions in mind

### Q7.3: Error Handling Philosophy
**Decision:** Three-tier hybrid strategy

**CRITICAL (surface immediately):**
- State corruption
- Data-losing merge conflicts
- Permission errors blocking core functions

**DEGRADED (pending work item + log):**
- Sync failures (network)
- Auto-checkpoint crashes
- MCP connection issues

**EXPECTED (silent log):**
- No git remote (sync disabled)
- Working tree dirty (skip sync)
- Rate-limited checkpoints

**User visibility:**
```bash
holistic status
✓ Session active: Phase 1.5 planning
✓ Last checkpoint: 2 minutes ago
⚠️  2 warnings:
  - Sync failing (network down) - will retry
  - Auto-checkpoint rate limited
```

### Q7.4: Cross-Device Sync Awareness
**Decision:** Smart auto-fetch with transparency

**Flow:**
```bash
holistic resume

Checking for updates from other devices...
📥 Found 3 new checkpoints from Machine A (2 minutes ago)
   Merging latest state...
   
📋 Resumed: Phase 1.5 planning
   New since last connect:
     • Checkpoint: "Added auto-tuning"
     • Checkpoint: "Implemented worktree sync"
     • Checkpoint: "Fixed merge conflict handling"
```

**Sync in progress:**
```
⏳ Machine A is syncing... waiting up to 5 seconds
✓ Sync complete, merged latest state
```

## Summary

**Key Principles Confirmed:**
- ✅ Workflow disappearance (automation by default)
- ✅ Low-touch configuration (smart defaults + optional tuning)
- ✅ Conservative automation (helpful, not invasive)
- ✅ Transparent operations (user can see what's happening)
- ✅ Easy escape hatches (disable features that don't fit)
- ✅ Cross-device first (sync built-in, not bolted-on)
- ✅ Repo-first architecture (works without daemon)
